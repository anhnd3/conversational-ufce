# Thesis Subsection Draft

## Paper-to-Implementation Discrepancies

This subsection presents a static forensic audit of the published UFCE journal paper against the author code snapshot distributed in `ufce/core_author`. The purpose of the audit is not to rerun the experiments or dispute the reported empirical outcomes by reproduction. Instead, the goal is to determine whether the public repository, as inspected statically, implements the same algorithmic and metric contracts described in the paper. The audited evidence base is limited to the code files on the observable generation and evaluation path and the published journal article identified by DOI `10.1007/s44196-024-00508-6`.

The audit shows that the repository preserves the broad architectural intent of UFCE, namely user-guided counterfactual generation, MI-guided multi-feature perturbations, neighborhood restriction to the desired class, and post-hoc evaluation with plausibility and feasibility-style metrics. However, the implementation diverges from the paper at several critical points. The largest inconsistencies concern the perturbation search itself, the distance geometry used to select counterfactuals, and the semantics of actionability and feasibility in the evaluation pipeline.

## Metric Contract Violations

The paper defines categorical proximity as a Jaccard-based distance and numerical proximity as a MAD-weighted Euclidean distance normalized to the interval `[0, 1]`, then combines them through a weighted mixed-distance rule in Algorithm 5. The observable repository path does not implement this contract directly. Candidate selection is performed through a helper that chooses the row with minimum Euclidean distance over continuous features only, ignoring categorical differences and the paper's `lambda` balancing term. The repository also defines multiple alternative distance helpers, but the active defaults use raw Euclidean distance for continuous proximity, while the optional MAD path is implemented as MAD-scaled cityblock distance rather than MAD-weighted Euclidean distance. Consequently, the "nearest" counterfactual returned by the code is not guaranteed to be the `z*` defined in the paper, and reported proximity values should not be treated as equation-equivalent to the published metric definitions.

## Validity Enforcement Gap

The paper's perturbation algorithms repeatedly state that a candidate is accepted only when it reaches the desired outcome and is plausible. The static code inspection does not support that claim for the active path. `Single_F` never checks plausibility during search. `Double_F` contains an unconditional append in its numeric-numeric branch, which allows non-flipping candidates to enter the returned counterfactual set. `Triple_F` reproduces the same pattern in several branches. Although the repository contains helpers for validity filtering and LOF-based plausibility, those checks are not part of the active search loop exercised by the observable experiment path. This matters because later feasibility reporting assumes that incoming candidate sets are already valid, yet feasibility itself does not re-verify `f(z) = t` before counting a row as feasible. The result is a validity-enforcement gap between the algorithmic contract described in the paper and the candidate-acceptance logic visible in the repository.

## Constraint Handling Differences

The paper frames user feedback as a feasible interval map with lower and upper bounds for each feature, and it explicitly describes protected features as part of the UFCE input contract. In the repository snapshot, user feedback is represented instead as a one-sided positive delta. Interval construction sets the lower bound to the factual value and clips only the upper bound against the neighborhood maximum. The lower-bound validation step described in the paper is absent. In parallel, all default dataset helpers populate `protectf` with an empty list, so immutable-feature protection exists in function signatures but is inactive in the default configuration. These differences weaken the claim that the searched subspace and immutable-feature guarantees in the public code are identical to those described in the article.

## Evaluation Semantics And Baseline Fairness

The paper defines actionability as the average number or percentage of changes drawn from the user-specified feature list and feasibility as the conjunction of validity, plausibility, and an actionability threshold. In the repository, actionability and feasibility are operationalized through fixed count thresholds rather than the paper's ratio-based description. The active `Actionability` reporting path returns row counts instead of the published average or percentage semantics. `Feasibility` uses LOF together with a hard-coded minimum number of actionable features and does not re-check validity. The baseline wrappers also apply constraints asymmetrically: basic DiCE receives a feature whitelist but no permitted ranges, DiCE-UF hardcodes Graduate-only ranges, and AR ignores the user-feedback arguments it is passed and is effectively post-hoc filtered only. This asymmetry weakens baseline fairness and complicates any direct interpretation of the repository's evaluation pipeline as a faithful implementation of the paper's comparative protocol.

## Omitted And Weakened Theory Elements

Some theoretical elements are present only partially or indirectly in the code. The repository contains a joint-proximity helper consistent with the paper's mixed-distance narrative, but the observable experiment script never invokes it. The repository also includes both active and secondary LOF helpers with different hyperparameters, which weakens traceability even though the paper does not publish a fixed LOF parameter. At the orchestration level, the visible experiment script in the audited snapshot is configured for the Graduate dataset and LR only, while the paper reports five datasets and discusses both LR and MLP. In addition, the active training helper uses 10-fold cross-validation, whereas the paper states fivefold cross-validation for Table 3. These differences do not by themselves invalidate the paper's empirical claims, but they do show that the public snapshot is not a decision-complete, line-by-line implementation of the published theoretical and evaluation contracts.

## Closeout Statement

Taken together, the findings support a cautious thesis interpretation: the public UFCE repository captures the high-level idea of user-feedback-guided counterfactual search, but it does not fully preserve the paper's formal search rules, distance geometry, or evaluation semantics. This gap is sufficient to justify treating the published repository as an audited baseline rather than as an implementation that can be accepted at face value without forensic review.
