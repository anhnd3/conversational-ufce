# Final Evidence Policy

This document defines which outputs are authoritative, which are diagnostic only, and how evidence should be interpreted for thesis review.

## 1. Authoritative Outputs (Primary Evidence)

These are the definitive evidence files generated by final wrappers. Reviewers can trust these as the primary record of results.

### Part I Authoritative Outputs

| Wrapper | Primary Output File | Purpose |
|---|---|---|
| `part1/01_reproduce_full_table7.py` | `<run_id>/summary.json` + dataset CSVs under `<out_dir>` | Full Table 7 reproduction numbers, rerunnable at any time. |
| `part1/01b_reproduce_ufce_only.py` | `<run_id>/summary.json` + per-dataset CSVs | UFCE-only behavior without baseline methods. |
| `part1/02_tune_final_parameters.py` | `<run_id>/summary.json` with stage-level results | Confirms tuned parameters produce expected metric improvements. |
| `part1/03_force_flip_audit.py` | `<run_id>/summary.json` (with null fields if not exposed by harness) | Documents raw candidate vs strict flip-valid discrepancy per dataset. |
| `part1/03b_blackbox_regression_audit.py` | `<run_id>/summary.json` | Validates model bundle assumptions against actual predictions. |
| `part1/04_parameter_bundle_ablation.py` | `<run_id>/summary.json` with ablation breakdowns | Supports thesis claim about coupled parameters (uf/f2change/step). |

### Part II Authoritative Outputs

| Wrapper | Primary Output File | Purpose |
|---|---|---|
| `part2/99_part2_closeout.py` | `<run_id>/summary.json`, evidence index JSON and Markdown | Aggregates all Part II evaluation gates. If this passes, the system is considered fully evaluated. |
| Individual Part II wrappers (01–06) | Per-report summary JSON under respective output roots | Detailed breakdown of each metric category for deeper inspection. |

### Product Authoritative Outputs

| Wrapper | Primary Output File | Purpose |
|---|---|---|
| `product/03_product_acceptance.py` | `<out_dir>/acceptance_summary.json`, Markdown report | MVP acceptance verdict combining automated smoke and manual session validation. |

## 2. Diagnostic-Only Outputs (Not Authoritative)

These outputs exist to support debugging and investigation but should not be cited as evidence:

- **Trace harness outputs** (`part1/05_trace_harness.py`): trace JSONL files are diagnostic, used for isolating specific suspicious queries. Not intended as primary evidence.
- **Intermediate CSVs**: raw output from reproduction scripts (e.g., per-fold metrics) unless aggregated by a closeout summary.
- **Log files** (`stdout.log`, `stderr.log`) under run directories: captured for debugging but not authoritative results.
- **Historical phase closeouts** (`scripts/run_phase1_closeout_suite.py`, etc.): kept for traceability, but superseded by current final wrappers.

## 3. Frozen Corpora

The following corpora are frozen and must not be regenerated without explicit approval:

| Corpus | Location | Purpose |
|---|---|---|
| Part II Tier A annotations (v1) | `docs/validation/corpora/part2_tier_a_bank_annotations_v1.json` | Annotation scoring ground truth. |
| Part II Tier B sessions (v1, v2 synth300) | `docs/validation/corpora/part2_tier_b_*` | Session replay corpora for conversation metrics. |
| Part II golden parity traces | `llm/models/bank/...` or model bundles directory | Frozen traces used by backend comparison runner. |
| Parser benchmark YAML (v1) | `llm_eval/benchmarks/ufce_bank_cf_parser_benchmark_v1.yaml` | Parser evaluation contract with expected outputs. |

**Regeneration is only allowed via:**
- `.venv/bin/python scripts/freeze_part2_bank_synth_corpora.py` for synthetic corpora, which overwrites specific v2 files explicitly.
- No other script may silently modify frozen corpora under `docs/validation/`.

## 4. Generated Output Files Not Committed to Git

The following generated outputs are **never committed** to the repository:

| Pattern | Rationale |
|---|---|
| `outputs/**` (except lightweight snapshots explicitly approved) | All run outputs, including summary JSON, Markdown summaries, trace files, CSVs. |
| SQLite DB runtime files (`*.sqlite3`) | Product session database; generated during demo/testing. |
| Large CSV intermediate files | Temporary analysis files from reproduction scripts. |
| Model cache (`.cache/`, `__pycache__/`) | Python bytecode and external model caches. |

The `.gitignore` must include these patterns. If you need to share evidence with a reviewer, use the evidence pack exporter:

```bash
python scripts/final/thesis/export_evidence_pack.py --part1-closeout <path> --part2-closeout <path>
```

## 5. Interpreting Warnings

Final wrappers produce `summary.json` files that may contain a `"warnings"` array:

- **"raw_candidate_count is null"** — The underlying trace harness does not expose this field; the wrapper cannot fabricate it. This is documented in `part1/03_force_flip_audit.py`.
- **LM Studio preflight failure** — Part II closeout will record "prefail" with a specific reason (connection refused, model not found). Do not treat this as passing.
- **"No evidence found for <dataset>"** — Indicates the dataset may be missing data files or fold definitions.

Warnings do not make an output invalid; they explain limitations of the current implementation. A summary can still carry status `"warning"` and remain authoritative, provided the core metrics are present.

## 6. Provenance Requirements

Every final wrapper output must include provenance metadata:
- `git_commit`: full SHA1 of the HEAD at run time
- `timestamp_local` + `timezone`: when the script ran
- `python_executable` and `python_version`: interpreter used
- `command_line`: exact command line invoked
- All input file paths that were read

If any of these fields is missing from a generated summary, the output's provenance is incomplete and should be flagged in review.
